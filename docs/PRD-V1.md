# MuseForge Studio 产品计划

## 1. 产品定位

MuseForge Studio 是面向二次元创作者的本地优先创作客户端。它不替代 A1111 WebUI、Aki WebUI 或 ComfyUI，而是在它们之上提供更适合产粮流程的资产管理、提示词辅助、配方沉淀、批量试验和任务编排。

产品默认连接用户自己的本地组件：

- A1111 / Aki WebUI：当前主要生图引擎。
- Ollama 或 OpenAI-compatible LLM：当前主要提示词与标签助手。
- 本地模型、LoRA、VAE、ControlNet 资源：由用户自行准备和管理。

后续会扩展到：

- ComfyUI 工作流。
- 云端 LLM、图像或视频 API。
- LoRA 训练工作流。
- Wan 等视频模型，用于动态壁纸和短视频素材。

## 2. 核心判断

WebUI 已经能完成专业生图。MuseForge 的价值不是把 WebUI 重新做一遍，而是解决二次元创作者反复遇到的工作流问题：

- 新手不会把中文需求写成稳定的 tag prompt。
- 产粮者需要保存好用的模型、VAE、LoRA、负面词和参数组合。
- 多模型、多 LoRA、多权重、多 seed 的试验结果难以比较和复用。
- 角色 IP、画风、姿势参考、ControlNet 配置没有统一资产抽象。
- LoRA 训练与使用割裂，用户不知道一个角色资产后续该如何持续使用。
- 本地组件分散，A1111、Ollama、模型目录、输出目录和任务记录缺少统一客户端。

## 3. 内容边界

MuseForge 不限制、不改写、不净化用户输入的提示词。用户对自己的模型、素材、提示词、输出结果、发布行为和合规风险负责。

产品也不提供 NSFW 或高风险内容的创作模板、引导、优化建议或专门辅助能力。这里的边界不是审查系统，而是产品能力设计边界。

## 4. 用户与场景

### 4.1 新手用户

用户配置 Ollama、WebUI 或后续 ComfyUI 后，用中文描述生成二次元图片。系统帮助补全 tag prompt、负面词和基础参数，生成前允许用户确认和编辑。

### 4.2 产粮用户

用户围绕某个角色、画风或主题批量抽卡，筛选好图，固定 seed，复用配方，比较模型、VAE、LoRA 权重和提示词差异。

### 4.3 角色 IP 用户

用户把自己喜欢的角色、OC 或参考资产沉淀成 Character IP。一个 IP 可以关联一个或多个 LoRA、触发词、推荐底模、权重、参考图、姿势和常用配方。

### 4.4 工作流用户

用户用 ControlNet、参考图、局部重绘、图生图或未来 ComfyUI 工作流来稳定控制姿势、构图、服装和系列图。

### 4.5 视频与动态壁纸用户

后续用户可以把已有角色、配方和图片结果继续送入视频模型，生成动态壁纸、短循环视频或轻量短片素材。

## 5. 产品模式

### 5.1 生图 Agent

Agent 是“创作助理”，不是黑箱自动画图器。它负责理解用户目标、读取本地资源、搜索 prompt tag、生成可编辑方案，并解释为什么这样安排。

当前阶段 Agent 能力应保持克制：

- 读取 prompt-all-in-one 标签库。
- 结合本地模型、LoRA、VAE 和 sampler 状态生成单次 txt2img 方案。
- 不默认开启 Hires Fix、Extras、ADetailer 或视觉评分。
- 不自动提交多轮重试。
- 对无触发词 LoRA 不编造触发词。

未来 Agent v2 再扩展：

- 读取用户资产和历史配方。
- 选择本地或云端引擎。
- 根据目标自动生成批量试验矩阵。
- 给出下一轮修改建议，但生成前仍让用户确认。

### 5.2 工作台

工作台是高级但易用的生产界面，面向高频产粮。它不是 WebUI 参数页复刻，而是围绕“配方、批量、对比、复用”组织。

工作台应优先支持：

- Recipe 管理：checkpoint、VAE、LoRA、prompt、negative、sampler、steps、CFG、seed、尺寸。
- 批量生成：一次生成 4/8/16 张，支持固定 seed 或随机 seed。
- 试验矩阵：比较 LoRA 权重、模型、VAE、prompt 片段。
- 结果筛选：收藏、淘汰、评分、备注、打开文件位置。
- 一键复用：从任意结果恢复生成配置。
- 资产引用：角色 IP、画风、姿势、ControlNet preset。

## 6. 资产体系

MuseForge 的长期核心是资产，而不是单次 prompt。

| 资产 | 含义 | 关键字段 |
| --- | --- | --- |
| Character IP | 角色或 OC 的可复用创作对象 | 名称、参考图、LoRA、触发词、推荐底模、权重、常用服装、常用负面词 |
| Style Asset | 画风或作者风格资产 | LoRA、prompt preset、推荐模型、VAE、权重范围 |
| Pose / Control Asset | 姿势、构图、参考控制 | ControlNet 类型、预处理器、模型、参考图、权重 |
| Prompt Preset | 可复用提示词片段 | 正面 tag、负面 tag、用途、语言、来源 |
| Recipe | 一套可复现生图配置 | 模型、VAE、LoRA、prompt、参数、seed、尺寸 |
| Generation Result | 一次生成结果 | 图片路径、缩略图、recipe 快照、评分、备注、父任务 |
| Video Template | 后续视频生成模板 | 角色、首帧、动作描述、视频模型、时长、分辨率 |

## 7. 当前已有能力

当前项目已经具备以下基础：

- React/Vite 前端原型，包含对话作图、AI 辅助作图、资源与设置相关界面。
- Node 后端 API，默认运行在 `http://127.0.0.1:8787`。
- A1111 / Aki WebUI 状态检查、模型扫描和 txt2img 调用。
- prompt-all-in-one tag 搜索接口。
- 基于 LLM 的 tag 生图方案生成。
- 单次 txt2img 任务队列、任务状态、取消、重试。
- 生成结果保存、列表、删除、打开文件位置。
- Provider 配置、测试、启用和本地密钥保存。
- Ollama 本地模型状态、安装、拉取和删除接口。
- checkpoint、LoRA、VAE、ControlNet、sampler 等资源索引。
- 资源 profile、用途标记和基础兼容性校验。
- ControlNet 资源导入和 preset 基础接口。
- LoRA 项目、素材导入、caption 更新、训练计划和训练任务雏形。

## 8. 主要差距

当前产品离真正可用的二次元产粮客户端还差这些能力：

- Recipe 还不是一等对象，用户难以保存和复用好配置。
- 批量抽卡、结果对比、收藏淘汰和筛图流程不完整。
- 试验矩阵缺失，无法系统比较模型、VAE、LoRA 权重和提示词差异。
- Character IP 与 Style Asset 还没有产品级抽象。
- 工作台仍偏参数展示，没有围绕“配方生产”重组。
- Agent 主要能生成 tag prompt，还不能引用资产、历史结果和配方库。
- ControlNet 还停留在资源层，缺少姿势/参考控制的用户工作流。
- LoRA 训练流程还没有形成“创建 IP -> 准备素材 -> 训练 -> 试画 -> 入库”的闭环。
- ComfyUI、云端引擎和视频模型尚未接入。
- 新手安装与环境修复仍需要更清楚的客户端体验。

## 9. 路线图

### P0：收敛现状

目标：稳定当前 A1111 单次生图、tag 辅助和文档方向。

- 移除旧高清 Agent、视觉评分和复杂动画叙事。
- 保留 prompt tag 辅助。
- 更新产品文档、架构文档和 API 文档。
- 明确 MuseForge 不是 WebUI 替代品。

### P1：Recipe 与批量抽卡

目标：让用户能保存一套好配置，并用它稳定批量出图。

- 新增 Recipe schema 与 CRUD。
- 从 plan、任务和生成结果保存 recipe 快照。
- 支持 4/8/16 张批量生成。
- 支持固定 seed、随机 seed、seed 递增。
- 画廊支持收藏、淘汰、备注、按 recipe 分组。

### P2：对比与试验矩阵

目标：让用户能系统调参，而不是凭感觉反复试。

- 支持 LoRA 权重矩阵。
- 支持 checkpoint / VAE 对比。
- 支持 prompt 片段开关对比。
- 支持同一 seed 下的横向比较。
- 生成结果展示差异来源。

### P3：角色 IP 与画风资产

目标：把 LoRA、触发词、参考图和常用配方封装成创作对象。

- Character IP 管理页。
- Style Asset 管理页。
- 资产与 Recipe 互相引用。
- Agent 可以读取资产并生成方案。

### P4：LoRA 制作工作流

目标：把炼制 LoRA 变成 IP 创建流程的一部分。

- 素材导入、去重、质量检查。
- 自动或半自动 caption。
- 训练参数推荐。
- 训练任务、日志和产物安装。
- 试画评估与入库。

### P5：ControlNet 与参考控制

目标：让用户能控制姿势、构图和参考图，而不必理解全部 ControlNet 细节。

- 姿势库和参考图库。
- OpenPose / Lineart / Canny / Depth 等 preset。
- 参考控制和 Recipe 绑定。
- 图生图、局部重绘和参考图创作入口。

### P6：Generation Agent v2

目标：Agent 能围绕用户资产和历史结果做可确认的创作规划。

- 读取 Recipe、IP、Style、Pose、历史结果。
- 自动选择本地或云端引擎。
- 生成批量试验方案。
- 输出可编辑计划和解释。
- 不自动绕过用户确认。

### P7：ComfyUI 与云端引擎

目标：支持更复杂工作流和非本地图像服务。

- ComfyUI 工作流模板导入。
- 云端图像 API adapter。
- 云端任务与本地任务统一进入画廊。
- 成本、速度、隐私和质量提示。

### P8：视频与动态壁纸

目标：让图片资产自然延伸到动态内容。

- Wan 等视频模型接入。
- 首帧/尾帧、角色、动作和镜头模板。
- 动态壁纸导出 preset。

## 10. 下一阶段计划

下一阶段建议用 2 到 3 周完成 P1 的最小闭环。

### 第 1 周：Recipe 数据层

- 定义 `Recipe`、`RecipeSnapshot`、`GenerationGroup` schema。
- 新增 Recipe CRUD API。
- 生成任务保存 recipe 快照。
- 生成结果可以回溯到 recipe。

验收：

- 用户可以把当前 plan 保存为 recipe。
- 用户可以从 recipe 重新生成一张图。

### 第 2 周：批量与画廊

- 支持 batch group，一次创建多张任务。
- 支持固定 seed、随机 seed、seed 递增。
- 画廊按 recipe / batch 分组。
- 支持收藏、淘汰、备注。

验收：

- 用户能用同一 recipe 生成 4/8 张图。
- 用户能收藏最好结果，并从该结果恢复参数。

### 第 3 周：工作台 UI 重组

- 工作台左侧为 recipe 和资产选择。
- 中间为批量生成与结果对比。
- 右侧为 prompt、参数和资源细节。
- Agent 只作为 tag 和方案助手出现，不占据主流程。

验收：

- 用户不用进入 WebUI，也能完成“选 recipe -> 批量出图 -> 筛图 -> 保存最佳 recipe”的完整流程。

## 11. 近期验收标准

P1 完成时，产品应满足：

- 现有单次 txt2img 正常可用。
- Agent 能继续读取 prompt tag 并生成更好的 tag prompt。
- 用户能保存、编辑、删除和复用 Recipe。
- 用户能从一个 Recipe 批量生成多张图。
- 用户能收藏、淘汰、备注生成结果。
- 用户能从任意生成结果恢复参数再次生成。
- 文档能清楚说明当前已有能力、主要差距、路线图和下一阶段任务。
